# Plan Brainstorm

機能ブレインストーミング → 方向性探索 → 仕様策定 → レビュー。
対話を通じて仕様書を完成させる。

## 原則

- **Spec is Truth**: 仕様書（`docs/specs/[name].md`）が成果物。テキスト議論だけで終わらない
- **Intent First**: 「誰が・何を・なぜ」を最初に明確にする
- **実現可能性ファースト**: 技術的に実現可能かを早い段階で検証する
- **対話的に進める**: 各フェーズでユーザーの確認を取りながら進行する

## 前提知識

以下のリファレンスを必要に応じて参照する:

- 共通思考原則 → [shared/thinking-principles.md](../shared/references/thinking-principles.md)
- プランニングクラフト原則 → [references/planning-craft.md](references/planning-craft.md)
- プロダクト思想 → `.design-intent/purpose.md`（存在する場合）

## Phase 1: Context — 何を・なぜ・誰のために

**このフェーズの目的**: 機能の背景と目的を明確にする土台を作る。

### 1.1 ヒアリング

**⚠️ ヒアリングの姿勢（重要）**

- ユーザーが「まだ形が見えていない」「思いつかない」と言っている場合、**選択肢を並べて「どれ？」と聞くのはNG**。まだ選べる段階ではない
- まず**オープンな質問**で背景・きっかけ・困りごとを引き出す（例：「その機能が欲しいと思ったきっかけや場面は？」）
- ユーザーの言葉から仮説を立てて**提案ベースで壁打ち**する。「こういうことかな？」→「違う／近い」のキャッチボールで形にしていく
- 構造化された選択肢（AskUserQuestion）は、**方向性が2〜3案に絞れた段階**で初めて使う

ユーザーのリクエストから以下を明確にする。不足があれば質問する:

**Who** — この機能を使う人は誰か

- 役割、スキルレベル、利用シーン

**What** — その人が達成したいこと

- 「機能を追加する」ではなく動詞で。「タスクの優先度を一目で把握して、今日やるべきことを決める」

**Why** — なぜ今これが必要か

- 現状の課題は何か。何が困っているか。どんなモチベーションがあるか

**Constraint** — 制約条件

- 技術的制約（使用スタック、既存コードとの整合）
- 時間的制約（期限、リソース）
- プロダクト制約（既存機能との一貫性）

### 1.2 プロダクト思想の確認

`.design-intent/purpose.md` が存在する場合、読み込んでContextとの整合を確認する。
存在しない場合はスキップ。

### 1.3 既存コードベースの調査

関連する既存コード・データ構造・コンポーネントを調査する:

```
- Kiri MCP: mcp__kiri__context_bundle() で関連コードを収集
- Serena MCP: mcp__serena__find_symbol() でシンボル検索
- Grep/Glob: 関連ファイルの検索
```

### 1.4 Context宣言

ヒアリング結果と調査結果をまとめてユーザーに提示し、合意を取る:

```
Context:
  Who: [具体的なペルソナ]
  What: [達成したいこと（動詞で）]
  Why: [なぜ今必要か（課題・モチベーション）]
  Constraint: [技術的・時間的・プロダクト的制約]
  既存資産: [関連する既存コード・データ構造]
```

**ユーザーの合意を得てから次に進む。**

## Phase 2: Explore — 方向性の探索と実現可能性検証

**このフェーズの目的**: Contextを満たす方向性を2〜3案提示し、技術的実現可能性を検証する。

### 2.1 ドメイン探索

共通思考原則に従い、プロダクトの世界を探索する:

**Domain** — この機能に関わる概念・メタファー・語彙（5つ以上）
**Defaults** — この種の機能で「よくある」実装パターン3つ（＝安易に選びがちなもの）
**Signature** — このプロダクトならではの特徴。他との差別化ポイント

### 2.2 実現可能性調査

既存のコードベース・データ構造を詳細に調査:

- 既存のデータスキーマ（Firestore等）との整合
- 既存のコンポーネント・フックの再利用可能性
- 外部依存（API、ライブラリ等）の有無と信頼性
- パフォーマンスへの影響

### 2.3 方向性提案

2〜3案を以下の形式で提示:

```
--- アプローチA: [名前] ---
概要: [1〜2文で方向性を説明]
ユーザー価値: [高/中/低 + 理由]
実装コスト: [高/中/低 + 理由]
保守性: [高/中/低 + 理由]
既存との整合: [高/中/低 + 理由]
リスク: [あれば具体的に]
推奨度: [★☆☆ / ★★☆ / ★★★]
Contextとの整合: [Phase 1のContextをどう満たすか]
```

**ユーザーに方向性を選んでもらう。ハイブリッドもOK。**

## Phase 3: Define — 仕様策定

**このフェーズの目的**: 選ばれた方向性を仕様書に落とし込む。

### 3.1 機能要件の詳細化

選択された方向性に基づいて:

- ユーザーストーリー（「〜として、〜したい。なぜなら〜」形式）
- 機能の一覧と優先度（Must / Should / Could）
- 画面・UI要素の一覧と遷移

### 3.2 データ構造設計

- 必要なデータモデル（スキーマ、型定義）
- 既存データ構造との関係
- データの流れ（CRUD操作、状態管理）

### 3.3 エッジケース・制約の洗い出し

- 異常系（エラー時の挙動、データ不整合）
- 境界条件（空データ、大量データ、同時操作）
- セキュリティ・権限
- アクセシビリティ（該当する場合）

### 3.4 スコープの明確化

- **Phase 1（Must）**: ないと機能として成立しないもの
- **Phase 2（Should）**: あると価値が上がるが、後からでも追加可能
- **Phase 3（Could）**: 将来的に検討する拡張
- **スコープ外**: 明示的にやらないこと

### 3.5 仕様書の生成

**雛形ファイル `docs/specs/_template.md` を参照して仕様書を作成する。**

出力先は機能の関わる範囲に応じてディレクトリを分ける:

```
docs/specs/
├── _template.md          ← 雛形（この形式に従う）
├── shared/               ← 全体に関わる機能
├── dashboard/            ← ダッシュボード固有
├── tasks/                ← タスク管理固有
└── settings/             ← 設定画面固有
```

**仕様書の構成（雛形に準拠）:**

| セクション         | 内容                                                         |
| ------------------ | ------------------------------------------------------------ |
| 背景・目的         | Who / What / Why / Constraint（Phase 1のContextから）        |
| 機能要件           | Must / Should / Could の3段階                                |
| データ構造         | TypeScript型定義 + Firestoreセキュリティルール（権限表付き） |
| 画面・UI           | mermaid状態遷移図 + ASCIIレイアウト + mermaidフローチャート  |
| エッジケース・制約 | 異常系、境界条件                                             |
| 非機能要件         | パフォーマンス、セキュリティ                                 |
| スコープ外         | 明示的にやらないこと                                         |
| 選定理由           | Phase 2で選んだアプローチとその理由                          |

**注意:**

- 状態遷移は mermaid `stateDiagram-v2` で記述する
- 操作フローの分岐は mermaid `flowchart` で記述する
- Firestoreセキュリティルールはコードだけでなく権限の表（本人/他ユーザー/未ログイン）も付ける

**ユーザーの合意を得てから次に進む。**

## Phase 4: Review — 3人のレビュワーによるチェック

**このフェーズの目的**: 策定者以外の3つの視点で仕様を検証する。

### 4.1 セルフチェック（自動）

仕様書を読み返し、以下をまず自分でチェック:

- Phase 1のContextとの整合
- 実装者が迷うポイントが残っていないか
- エッジケースの抜け漏れ
- スコープの妥当性

問題があれば修正してからレビュワーに渡す。

### 4.2 レビュワー3名によるレビュー

仕様書とPhase 1〜3の情報を基に、以下の3つの視点でレビューする:

#### テックリード

**視点**: 技術的妥当性・保守性

- 技術的に実現可能か。見落としている制約はないか
- 既存のアーキテクチャ・パターンと整合しているか
- パフォーマンス・セキュリティに問題はないか
- 将来の拡張を過度に阻害しないか
- データ構造は適切か

#### ユーザー代表

**視点**: ユーザー体験の完成度

- ユーザーの目的を達成できるか
- 操作フローに不自然なステップがないか
- エッジケースでユーザーが困る状況はないか
- 既存機能との一貫性はあるか
- 認知負荷は適切か

#### プロダクトオーナー

**視点**: プロダクト価値・Contextとの整合

- Phase 1で定めた「Who / What / Why」を満たしているか
- ビジネス価値を提供できるか
- スコープは適切か（広すぎ / 狭すぎ）
- 優先度の判断は正しいか
- プロダクト思想との整合（`.design-intent/purpose.md` がある場合）

### 4.3 レビュー結果の提示

```
Spec Review: [機能名]

■ テックリード
  ✅ [良い点]
  ⚠️ [改善提案] → [具体的な修正案]

■ ユーザー代表
  ✅ [良い点]
  ⚠️ [改善提案] → [具体的な修正案]

■ プロダクトオーナー
  ✅ [良い点]
  ⚠️ [改善提案] → [具体的な修正案]

→ 修正提案 N件。仕様書に反映する？それとも調整したい箇所がある？
```

### 4.4 修正ループ

ユーザーの判断に基づいて:

- **自動修正**: レビュー指摘を仕様書に反映 → 再確認
- **ユーザー調整**: ユーザーのフィードバックを反映して修正
- **完了**: 問題なければ仕様書を確定

修正ループは最大3回。3回で収束しない場合はユーザーに方針を相談する。

## コミュニケーションルール

- 各フェーズの開始時に「今からPhase Xをやるね」と一言伝える
- **質問時は必ず `AskUserQuestion` ツールを使う**（テキストだけの質問は避ける）
  - 予想される回答を選択肢（2〜4個）として提示する
  - 自由回答が必要な場合も、想定される選択肢を提示しつつOtherで対応
- 「なぜその案を推すか」を常に説明する
- レビュワーの指摘は客観的に。策定者（自分）の弁護はしない

## 中断・再開

途中で中断する場合:

- 現在のフェーズと状態をユーザーに伝える
- 仕様書が途中の場合はドラフト状態であることを明記する
- 再開時はPhase番号を指定して途中から再開可能
