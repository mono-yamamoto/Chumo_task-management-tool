機能企画・仕様策定に特化した原則。
plan-brainstormのPhase 2（Explore）とPhase 4（Review）で参照する。

共通思考原則は → [shared/thinking-principles.md](../../shared/references/thinking-principles.md)

---

## 1. 実現可能性ファースト

「やりたいこと」と「できること」のギャップを最初に把握する。

### 検証すべき観点

- **技術スタック**: 現在のスタックで実現できるか。新しいライブラリや技術が必要か
- **既存コードとの整合**: 既存のデータ構造・API・コンポーネントに沿えるか。大幅な変更が必要か
- **パフォーマンス**: データ量が増えた時に問題が出ないか
- **外部依存**: 外部API・サービスへの依存があるか。その信頼性・コストは

### 実現可能性の3段階

- **そのまま可能**: 既存の仕組みの上に乗るだけ
- **拡張が必要**: 既存の仕組みを拡張すれば対応可能（スキーマ追加、新コンポーネント等）
- **大幅変更が必要**: アーキテクチャレベルの変更が必要。コスト・リスクを明示する

## 2. スコーピングの原則

### MVP思考

- 「最小限で価値を届けられる範囲」を最初に定義する
- Phase 1（Must）→ Phase 2（Should）→ Phase 3（Could）で段階的に拡張
- 「ないと機能として成立しないもの」だけがPhase 1

### スコープの切り方

- **ユーザーストーリー単位**: 「ユーザーが〜できる」で切る。技術タスク単位で切らない
- **独立してリリース可能**: 各スコープが単体で意味を持つ
- **検証可能**: 各スコープの完了を客観的に判断できる

### スコープクリープの防止

- 仕様策定中に「ついでにこれも」が出たら、Phase 2/3に分類する
- 既存のスコープを広げるのではなく、新しいスコープとして切り出す

## 3. 仕様書の書き方

### 良い仕様書の条件

- **実装者が迷わない**: 「これはどうするんだろう？」が残らない
- **テスト可能**: 仕様から受け入れ条件が導ける
- **変更に開かれている**: 細部の実装手段まで固めすぎない。「何を」は明確に、「どうやって」は余白を残す

### 仕様書に含めるもの

1. **背景・目的**: なぜこの機能が必要か（Phase 1のContextから転記）
2. **機能要件**: ユーザーが何をできるようになるか（ユーザーストーリー形式推奨）
3. **データ構造**: どんなデータを扱うか（スキーマ、型定義）
4. **画面・UI**: どの画面で操作するか（画面一覧、遷移）
5. **エッジケース**: 異常系・境界条件
6. **非機能要件**: パフォーマンス、セキュリティ、アクセシビリティ（該当する場合）
7. **スコープ外**: 明示的に「やらないこと」を書く

### 仕様書に含めないもの

- 具体的な実装手順（エンジニアリングフェーズの領域）
- コードレベルの設計詳細（関数名、クラス設計等）
- テストケースの詳細（テスト計画はエンジニアリングで）

## 4. 方向性検討のフレームワーク

### アプローチ比較の軸

各案を比較する時に使う軸:

- **ユーザー価値**: ユーザーにとっての使いやすさ・分かりやすさ
- **実装コスト**: 実装の難易度・工数
- **保守性**: 将来の変更・拡張のしやすさ
- **既存との整合**: 既存の機能・UIパターンとの一貫性
- **リスク**: 技術的リスク、UXリスク

### 比較の出し方

```
--- アプローチA: [名前] ---
概要: [1〜2文]
ユーザー価値: [高/中/低 + 理由]
実装コスト: [高/中/低 + 理由]
保守性: [高/中/低 + 理由]
既存との整合: [高/中/低 + 理由]
リスク: [あれば具体的に]
推奨度: [★☆☆ / ★★☆ / ★★★]
```

## 5. プランニングアンチパターン

- **全部入り症候群**: 最初から完璧を目指して、Phase 1のスコープが膨れる
- **技術ドリブン**: 「この技術を使いたい」から始めて、ユーザー価値を後付けする
- **曖昧な要件**: 「いい感じにする」「使いやすくする」— 具体性がない
- **エッジケース後回し**: 正常系だけ考えて、異常系を「後で考える」にする
- **既存無視**: 既存のコード・パターン・データ構造を調べずに新しく設計する
- **過剰設計**: 将来の要件を想像して、今必要ない汎用性を盛り込む
- **スコープ外の未定義**: 「やらないこと」を明示しないせいで、認識がずれる

## 6. レビュー観点（Phase 4で使用）

### テックリード視点

- 技術的に実現可能か。見落としている制約はないか
- 既存のアーキテクチャ・パターンと整合しているか
- パフォーマンス・セキュリティに問題はないか
- 将来の拡張を過度に阻害しないか

### ユーザー視点

- ユーザーの目的を達成できるか
- 操作フローに不自然なステップがないか
- エッジケースでユーザーが困る状況はないか
- 既存機能との一貫性はあるか

### プロダクトオーナー視点

- ビジネス価値を提供できるか
- スコープは適切か（広すぎ/狭すぎ）
- 優先度は正しいか
- Phase 1のContextで定めた目的を満たしているか
